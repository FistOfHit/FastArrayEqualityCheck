# Empirical Evaluation

`FAE` now includes lightweight measurement tools so the fingerprint pipeline can
be evaluated with data instead of intuition alone. The measurement layer now
supports side-by-side algorithm comparisons.

## Included tools

- `fae_benchmark`: throughput-oriented measurements for fingerprint generation
  and comparison
- `fae_collision_study`: empirical collision study over several array lengths
  and input distributions

These are intentionally small standalone executables rather than a heavyweight
benchmark framework. The goal is portability and reproducibility inside the same
build system as the library.

## Compared algorithms

The current tooling compares:

- `legacy_fold`: the older public pipeline retained for compatibility and
  direct comparison
- `hash_combine`: the current default public pipeline, based on a simpler
  ordered hash-combine approach
- `cantor_mix`: a public opt-in alternative that uses Cantor pairing on 16-bit
  fragments before mixing them into a full fingerprint state
- `tuple_mix`: a public opt-in alternative that mixes adjacent value pairs
  before folding them into the fingerprint state

The public library API now exposes `legacy_fold`, `hash_combine`,
`cantor_mix`, and `tuple_mix`, with `hash_combine` as the default.

## Build

```bash
cmake -S . -B build -DFAE_BUILD_BENCHMARKS=ON
cmake --build build
```

## Run

```bash
./build/fae_benchmark
./build/fae_collision_study
```

Each tool also supports lightweight profiles:

```bash
./build/fae_benchmark --profile=smoke
./build/fae_collision_study --profile=smoke
./build/fae_benchmark --profile=deep
./build/fae_collision_study --profile=deep --seed=123456789
```

- `smoke`: reduced sizes or sample counts for CI and quick regression checks
- `default`: the baseline local profile used for checked-in summaries
- `deep`: a larger local run for investigating algorithm changes

## Throughput benchmark methodology

`fae_benchmark` reports simple wall-clock estimates for:

- `fingerprint`
- `compare_match`
- `compare_differ`

across multiple array sizes, representative types, and compared algorithms.

The output reports:

- algorithm name
- input size in elements
- iteration count used for the case
- approximate nanoseconds per call
- approximate MiB per second processed

These numbers are best used for trend analysis and regression detection. They
are not a substitute for careful microbenchmarking on pinned hardware.

## Collision study methodology

`fae_collision_study` generates many arrays and measures how often distinct
inputs land in the same `fae::Fingerprint`.

The study tracks:

- algorithm name
- `samples`: total generated inputs
- `unique_fp`: number of distinct fingerprints observed
- `duplicates`: repeated identical inputs generated by the study
- `collisions`: distinct inputs that produced the same fingerprint

The current study uses five distributions:

- `uniform_full_range`
- `low_range`
- `sparse_boolean`
- `monotonic_step`
- `paired_low_range`

The study is empirical, not a proof. A zero-collision run does not establish a
formal collision bound; it only says the sampled workload did not reveal one.

## CI usage

The CMake test suite includes reduced-profile measurement smoke tests when
`FAE_BUILD_BENCHMARKS=ON`. This keeps the benchmark and collision executables
buildable and runnable in CI without turning the CI job into a full empirical
campaign.

## How to use the results

- Record a baseline summary after meaningful algorithm changes.
- Keep checked-in snapshots under `docs/results/` for historical comparison.
- Compare runs before and after algorithm changes.
- Compare the current default pipeline against retained and experimental
  alternatives before changing the default again.
- Watch for throughput regressions when changing mixers or fold behavior.
- Watch for non-zero empirical collisions on small, dense distributions first.
- Treat benchmark and collision data as one input into design decisions, not as
  a replacement for API clarity or theoretical analysis.

See `docs/benchmark-baseline.md` for the stable entry point and
`docs/results/index.md` for the versioned archive.
